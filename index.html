    Duplicate Code

    Why my code is so long? It should be 50-100 lines, but this one is more than 500 lines?

    Oh I forgot that I copy and paste too much, but this is so strange, the code is

    running perfectly. Why this happened?

    That is because, there's same method, so the code still run the normal way.

    But, sometimes the method is easy to identify, so the code will be easily refactor.

    But again, sometimes the code has to be make in specifc to be easy to identify.

    The rule of three is when you already copy and paste for 3 times, it is the time

    you do a refactor code

    Example

              
            public class Calculation {
    
              public int add(int a, int b) {
                  return a + b;
              }
              
              public int subtract(int a, int b) {
                  return a - b;
              }
              
              public int multiply(int a, int b) {
                  return a * b;
              }
              
              public int addArrayElements(int[] array) {
                  int sum = 0;
                  for (int i = 0; i < array.length; i++) {
                      sum += array[i];
                  }
                  return sum;
              }
              
              public int multiplyArrayElements(int[] array) {
                  int product = 1;
                  for (int i = 0; i < array.length; i++) {
                      product *= array[i];
                  }
                  return product;
              }
              
              // Duplicate code starts here
              public double calculateAverage(int[] array) {
                  double sum = 0;
                  for (int i = 0; i < array.length; i++) {
                      sum += array[i];
                  }
                  return sum / array.length;
              }
              
              public int findMax(int[] array) {
                  int max = array[0];
                  for (int i = 1; i < array.length; i++) {
                      if (array[i] > max) {
                          max = array[i];
                      }
                  }
                  return max;
              }
              // Duplicate code ends here
          }

    In the above code, you can see that there is a repetition of code for

    iterating over an array and performing some operation (sum or product)

    on its elements. This repetition violates the DRY (Don't Repeat Yourself)

    principle, which is a fundamental principle in software development. This

    duplication can lead to maintenance issues because if you need to change

    the logic of iterating over an array, you would have to do it in multiple

    places, increasing the chances of introducing bugs and making

    the code harder to maintain.

    Lazy Class

    This is neat and clean code, but there are so many classes. That make me so confused

    Hmmm... Is this bad?

    Well yeah, make so many classes in one code can make you confused sometimes.

    And also sometimes make one classes with too many components, is not good too

    That's why we make a different class. But also at the sametime, too many classes

    with just one function is also useless and not eficient, can make other proggrammer

    get confused when they try to understand your code, and it's call lazy class.

    When there's too many classes but with just one function. Lazy class can also

    happened when you are refactoring the code.

    Example

              
            public class Employee {
              private String name;
              private String id;
              
              public Employee(String name, String id) {
                  this.name = name;
                  this.id = id;
              }
              
              // Getters and setters
              public String getName() {
                  return name;
              }
              
              public void setName(String name) {
                  this.name = name;
              }
              
              public String getId() {
                  return id;
              }
              
              public void setId(String id) {
                  this.id = id;
              }
          }

    In this example, the Employee class only contains attributes and their corresponding

    getters and setters. It doesn't contain any behavior or logic specific to an employee

    such as methods for calculating salaries, managing leaves, or performing other

    employee-related tasks. To improve this class, you might consider adding more functionality

    that is specific to an employee, such as methods for calculating salaries, managing leaves,

    or performing other employee-related tasks. Otherwise, if the class remains as it is, it can

    be considered a "lazy class" because it doesn't do enough to justify its existence.

    Dead Code

    HUAAA... I already make a really long code, but I don't know what's meaning of this code

    What is this for? Is this really usefull for the program? But when the program is running,

    the code that I already make is not executed by the program. Damn... :')

    That's call "Dead Code", when the code is not used or executed by your application.

    This bad code smell can happends when we are developing process

    Example

              
            public class Calculator {
    
              // This method is never called
              public int add(int a, int b) {
                  // Dead code
                  int result = a + b;
                  return result;
              }
              
              // This method is never called
              public int subtract(int a, int b) {
                  // Dead code
                  int result = a - b;
                  return result;
              }
              
              public int multiply(int a, int b) {
                  // This method is called
                  int result = a * b;
                  return result;
              }
              
              public int divide(int a, int b) {
                  // This method is called
                  if (b == 0) {
                      throw new IllegalArgumentException("Cannot divide by zero");
                  }
                  int result = a / b;
                  return result;
              }
          }

    In the example above, the add and subtract methods are never called. Therefore,

    you can safely remove them from the Calculator class.

    As the codebase evolves, methods or blocks of code that were once necessary may become obsolete.

    Solution

              
            public class Calculator {
    
              public int multiply(int a, int b) {
                  // This method is called
                  int result = a * b;
                  return result;
              }
              
              public int divide(int a, int b) {
                  // This method is called
                  if (b == 0) {
                      throw new IllegalArgumentException("Cannot divide by zero");
                  }
                  int result = a / b;
                  return result;
              }
          }

    By removing dead code, you reduce clutter in the codebase and make it easier

    for developers to understand and maintain the application.

    Speculative Generality

    Hmm... Do I need this code in the future?? I'm not sure this will be useful.

    But what if I need later?

    Well yes, usually a lot of proggrammers, make class, method, field or parameter even

    though this is not in the requirement. But yes a lot of proggramers just make this for

    their own speculation, and they are not really sure if this is need in the future.

    So this is call speculative generality, this is can become a problem, because the code

    that are not useful instead of deleted it, can make other people confused when they are

    trying to read it. This is remind me of people with anxiety, is like same sitution with

    this bad code

    Example

              
            public interface Animal {
              void eat();
              void sleep();
              void makeSound();
          }
          
          public class Dog implements Animal {
              @Override
              public void eat() {
                  System.out.println("Dog is eating");
              }
          
              @Override
              public void sleep() {
                  System.out.println("Dog is sleeping");
              }
          
              @Override
              public void makeSound() {
                  System.out.println("Bark");
              }
          }
          
          public class Cat implements Animal {
              @Override
              public void eat() {
                  System.out.println("Cat is eating");
              }
          
              @Override
              public void sleep() {
                  System.out.println("Cat is sleeping");
              }
          
              @Override
              public void makeSound() {
                  System.out.println("Meow");
              }
          }
          
          public class AnimalApp {
              public static void main(String[] args) {
                  Animal dog = new Dog();
                  dog.eat();
                  dog.sleep();
                  dog.makeSound();
          
                  Animal cat = new Cat();
                  cat.eat();
                  cat.sleep();
                  cat.makeSound();
              }
          }

    In this example, we have an Animal interface with methods eat(), sleep(), and

    makeSound(). We then have concrete implementations for Dog and Cat. However,

    at the current stage, there's no apparent need for an Animal interface as both

    Dog and Cat classes provide similar functionality.

    Solution

    If the abstraction isn't currently needed or likely to be needed in the future,

    it's better to remove it to simplify the code. However, if there's a genuine need

    for abstraction, ensure that it's driven by real requirements and not just

    speculative thinking
